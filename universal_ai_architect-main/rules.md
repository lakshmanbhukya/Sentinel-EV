# 🛠️ Universal AI Architect: Development Rules & Guidelines

## Overview

This document establishes the comprehensive development rules and guidelines for all projects generated using the Universal AI Architect system. These rules ensure consistency, quality, and adherence to best practices across all generated applications.

## Core Principles

### 1. Follow All Universal Standards
All generated code and documentation MUST adhere to the standards defined in this repository:

- **Framework Rules** (`blueprints/framework-rules.md`): Technology stack, component architecture, and coding standards
- **Accessibility Standards** (`blueprints/accessibility-std.md`): WCAG compliance, semantic HTML, and inclusive design
- **Dashboard Patterns** (`blueprints/dashboard-patterns.md`): Layout hierarchy, UX patterns, and visual organization
- **Architect Protocol** (`protocols/architect-protocol.md`): Project provisioning and context generation process
- **Prompt Template** (`templates/PROMPT_TEMPLATE.md`): Master prompt structure for coding IDE integration

### 2. Technology Stack Compliance

#### Required Stack Components
- **Frontend Framework**: React 18+ with Vite for build tooling
- **Routing**: React Router DOM for client-side navigation
- **Styling**: Tailwind CSS with utility-first approach
- **Component Library**: Shadcn/UI (Radix UI + Tailwind) for accessible primitives
- **Icons**: Lucide React for consistent iconography
- **State Management**: Zustand for global state, TanStack Query for server state
- **TypeScript**: Strict mode enabled, no `any` types allowed

#### Database Requirements
- **Primary Database**: PostgreSQL for production applications
- **Development**: PostgreSQL with proper indexing and constraints
- **Migrations**: Structured migration system for schema changes
- **Connection Management**: Connection pooling for performance

#### AI Services Integration
- **Primary Provider**: GroqAI for all AI-powered features
- **Fallback Strategy**: Always implement graceful degradation
- **API Management**: Proper rate limiting and error handling

### 3. Project Structure Standards

All projects MUST follow this directory structure:

```
├── src/
│   ├── components/
│   │   ├── ui/              # Shadcn primitives
│   │   ├── shared/          # Reusable components
│   │   └── [feature]/       # Feature-specific components
│   ├── hooks/               # Custom React hooks
│   ├── lib/                 # Utility functions
│   ├── pages/               # Route components
│   ├── types/               # TypeScript definitions
│   ├── App.tsx              # Main app with routing
│   └── main.tsx             # Vite entry point
├── public/                  # Static assets
├── .brainchain/             # Context files (generated by Architect)
│   ├── data-model.md
│   ├── design-tokens.md
│   ├── ui-inventory.md
│   └── project-requirements.md
└── START_HERE_PROMPT.md     # Master prompt for implementation
```

### 4. Code Quality Requirements

#### Component Architecture
- **Atomic Design**: Small, focused, single-responsibility components
- **TypeScript Interfaces**: Every component must have proper prop types
- **Error Boundaries**: Implement error handling for all major components
- **Loading States**: All data-dependent components must show loading states

#### Styling Standards
- **Mobile-First**: Always code responsive design starting with mobile
- **No Arbitrary Values**: Use Tailwind's design system scales
- **Conditional Classes**: Use `cn()` utility for dynamic styling
- **Dark Mode**: Support dark mode with CSS variables
- **Accessibility**: Minimum WCAG 2.1 AA compliance

#### Performance Requirements
- **Bundle Size**: Optimize for minimal bundle size
- **Code Splitting**: Implement route-based code splitting
- **Lazy Loading**: Use React.lazy for non-critical components
- **Caching**: Implement appropriate caching strategies

### 5. Database Design Rules

#### Schema Requirements
- **UUID Primary Keys**: Use `gen_random_uuid()` for all primary keys
- **Proper Data Types**: Use appropriate PostgreSQL data types (DECIMAL for money, TIMESTAMP WITH TIME ZONE for dates)
- **Foreign Key Constraints**: Always implement proper relationships with CASCADE options
- **Indexing Strategy**: Create indexes for all query patterns

#### Data Integrity
- **ACID Compliance**: All transactions must be ACID compliant
- **Validation**: Implement both client-side and database-level validation
- **Audit Trails**: Include created_at and updated_at timestamps
- **Soft Deletes**: Consider soft deletes for important data

### 6. API Design Standards

#### REST API Requirements
- **RESTful Design**: Follow REST principles for all endpoints
- **HTTP Status Codes**: Use appropriate status codes (200, 201, 400, 401, 404, 409, 500)
- **Error Handling**: Consistent error response format
- **Rate Limiting**: Implement rate limiting for all public endpoints
- **Documentation**: OpenAPI/Swagger documentation for all endpoints

#### Real-time Features
- **WebSocket Support**: Use WebSockets for real-time updates
- **Graceful Degradation**: Fallback to polling if WebSockets fail
- **Connection Management**: Proper connection lifecycle management

### 7. Testing Requirements

#### Testing Strategy
- **Unit Tests**: Minimum 80% code coverage for business logic
- **Integration Tests**: Test API endpoints and database operations
- **Property-Based Tests**: Use fast-check for comprehensive validation
- **End-to-End Tests**: Test critical user flows

#### Testing Configuration
- **Framework**: Jest for unit tests, Playwright for E2E
- **Property Tests**: Minimum 100 iterations per property test
- **Mock Strategy**: Mock external APIs, test real database operations
- **CI/CD**: All tests must pass before deployment

### 8. Security Requirements

#### Authentication & Authorization
- **Secure Authentication**: Implement proper authentication mechanisms
- **Role-Based Access**: Implement RBAC where applicable
- **Session Management**: Secure session handling
- **Input Validation**: Validate all user inputs

#### Data Protection
- **Environment Variables**: Use environment variables for sensitive data
- **SQL Injection Prevention**: Use parameterized queries
- **XSS Prevention**: Sanitize all user-generated content
- **HTTPS Only**: All production traffic must use HTTPS

### 9. Accessibility Compliance

#### WCAG 2.1 AA Requirements
- **Contrast Ratios**: Minimum 4.5:1 for normal text
- **Keyboard Navigation**: Full keyboard accessibility
- **Screen Reader Support**: Proper ARIA labels and semantic HTML
- **Focus Management**: Visible focus indicators
- **Touch Targets**: Minimum 44x44 pixels for interactive elements

#### Implementation Standards
- **Semantic HTML**: Use proper HTML elements
- **Alt Text**: All images must have descriptive alt text
- **Form Labels**: All form inputs must have associated labels
- **Error Messages**: Clear, descriptive error messages

### 10. Performance Standards

#### Response Time Requirements
- **API Responses**: Maximum 2 seconds for data queries
- **Page Load**: Maximum 3 seconds for initial page load
- **Interactive**: Time to Interactive under 5 seconds
- **Database Queries**: Optimize all queries for sub-second response

#### Optimization Strategies
- **Image Optimization**: Compress and optimize all images
- **Code Splitting**: Implement route-based code splitting
- **Caching**: Use appropriate caching at all levels
- **CDN**: Use CDN for static assets

### 11. Documentation Requirements

#### Code Documentation
- **JSDoc Comments**: Document all complex functions
- **README Files**: Comprehensive setup and usage instructions
- **API Documentation**: Complete API documentation
- **Architecture Diagrams**: Visual system architecture documentation

#### User Documentation
- **User Guides**: Clear user interface documentation
- **Installation Guides**: Step-by-step setup instructions
- **Troubleshooting**: Common issues and solutions
- **Changelog**: Document all changes and updates

### 12. Deployment & DevOps

#### Environment Management
- **Environment Separation**: Clear separation of dev/staging/production
- **Configuration Management**: Environment-specific configurations
- **Secret Management**: Secure handling of sensitive data
- **Monitoring**: Comprehensive application monitoring

#### CI/CD Pipeline
- **Automated Testing**: All tests run automatically
- **Code Quality Checks**: Linting and formatting validation
- **Security Scanning**: Automated security vulnerability scanning
- **Deployment Automation**: Automated deployment processes
